# 发布管理器

本 Skill 用于把“代码写好了”变成“可以安全交付”。它不是所有模块默认都走完整发布流程；应先判断模块类型、交付等级和本次交付方式，再决定最低足够的发布与回滚要求。

## 使用时机

- 准备上线或交付用户
- 执行迁移、配置切换、重启或版本切换
- 需要发布前检查、Smoke Check 或回滚方案

## 使用方式

1. 先识别当前模块的模块类型（`页面类` / `API 类` / `数据处理类` / `工具类`）和交付档位（`exploratory` / `standard` / `high-risk`）
2. 判断这次是正式上线、灰度、内部交付、手工运行还是仅产出可复用工件
3. 使用下方模板生成或更新 `.ai-os/release-plan.md`
4. 仅检查当前发布真正涉及的构建、测试、迁移、配置、Feature Flag、环境变量、受影响服务和执行步骤
5. 使用 `create-ai-os release-check` 或同等检查方式验证发布计划完整性
6. 高风险发布必须明确人工审批；发布后记录结果；若异常，转入 `incident`

## 约束

- 不要把“代码已合并”误当成“已可交付”
- 不要对 `exploratory` 工具类或内部交付场景机械追加完整上线审批链
- 若项目声明了目标运行栈，不要用开发 fallback（如 H2 / SQLite / mock auth）替代正式运行态证据
- 本 Skill 负责发布准备和回滚要求，不替代事故处理和复盘

## 必查项

- 是否明确这次交付的目标环境、交付方式和受影响范围
- 是否明确目标运行态与开发 fallback 的区别，并确保正式交付证据来自目标运行态
- 是否定义了与模块类型匹配的 Smoke Check 或等价验证方式
- 是否具备最小必要观测性
- 是否识别高风险步骤和审批点
- 若涉及迁移、配置变更、重启、调度、安装或版本切换，是否存在对应回退路径

## 按模块类型检查发布要点

### 页面类

- 检查构建产物、静态资源、环境变量、路由可达性和关键页面 Smoke Check
- 若涉及后端依赖，补充 API 可用性与权限验证

### API 类

- 检查接口契约变更、调用方影响、配置变更、鉴权、限流和回滚方式
- 若存在破坏性变更，明确灰度或兼容窗口

### 数据处理类

- 检查触发方式、调度窗口、重跑策略、补数方案、幂等性和失败恢复
- 明确结果校验方式，不要只写“任务已执行”

### 工具类

- 检查安装 / 分发方式、版本切换、参数兼容性、帮助输出和失败退出行为
- 对仅内部使用的工具，不要默认要求完整上线审批链

## 按交付档位缩放发布要求

### exploratory

- 只有在用户明确要求正式交付时，才需要完整 `release-plan`
- 否则保留最小交付说明、运行方式和已知限制即可

### standard

- 需要明确交付步骤、Smoke Check、回退路径、负责人和关键风险

### high-risk

- 在 standard 基础上，必须明确审批点、回滚演练或等价准备、安全约束和跨服务影响

## 模板引用

- 发布计划：读取 `.agents/templates/project/release-plan.md` 作为模板生成 `.ai-os/release-plan.md`

### 示例：high-risk API 发布检查

- 输入：`.ai-os/release-plan.md`、受影响服务、迁移步骤、回滚路径
- 输出：补齐发布步骤、Smoke Check、审批点和高风险剩余项
- 约束：若破坏性变更缺少兼容窗口或回滚方案，则不得给出通过结论
