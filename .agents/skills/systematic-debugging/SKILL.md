# 系统化调试方法论

## 概述

盲目修复浪费时间并制造新 Bug。快速补丁只是掩盖根本问题。

**核心原则：修复之前，必须先找到根因。治标不治本等于失败。**

## 铁律

```
没有完成根因调查，不得提出修复方案
```

如果你没有完成第一阶段，你不能提出任何修复方案。

## 使用时机

适用于任何技术问题：
- 测试失败
- 生产环境 Bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况必须使用**：
- 时间紧迫（紧急情况更容易诱导猜测）
- "一个小修复"看起来很明显
- 你已经尝试了多次修复
- 上次修复没有奏效
- 你没有完全理解问题

**不要跳过：**
- 问题看起来简单（简单的 Bug 也有根因）
- 你很着急（系统化方法比胡乱猜测更快）

## 四个阶段

你必须**依次完成**每个阶段，才能进入下一阶段。

### 第一阶段：根因调查

**在尝试任何修复之前**：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 错误信息通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠地触发？
   - 确切的操作步骤是什么？
   - 每次都会发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变化可能导致了这个问题？
   - Git diff、最近的提交
   - 新增依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**

   当系统有多个组件（CI → 构建 → 签名，API → 服务 → 数据库）时：

   **在提出修复之前，先添加诊断信息**：
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置是否正确传递
     - 检查每一层的状态

   运行一次收集证据，确定在哪里断裂
   然后分析证据定位故障组件
   然后调查该特定组件
   ```

   **如果系统存在共享包装层或转换层，也要先把它们纳入证据链**：
   - request wrapper / interceptor
   - DTO / adapter / serializer
   - 路由 / 鉴权 / 中间件
   - 全局样式变量 / 主题基准

5. **追踪数据流**
   - 错误值从哪里来？
   - 谁用错误值调用了这里？
   - 一直向上追溯直到找到源头
   - 在源头修复，而不是在症状处

### 第二阶段：模式分析

**修复前先找到模式**：

1. **找到可工作的示例**
   - 在同一代码库中找到类似的正常工作的代码
   - 什么类似的东西是正常的？

2. **对照参考实现**
   - 如果在实现某个模式，**完整阅读**参考实现
   - 不要跳读，每一行都看
   - 完全理解模式后再应用

3. **识别差异**
   - 正常和异常之间有什么不同？
   - 列出每一个差异，无论多小
   - 不要假设"这不重要"

4. **理解依赖**
   - 需要什么其他组件？
   - 什么设置、配置、环境？
   - 做了什么假设？

### 第三阶段：假设与测试

**科学方法**：

1. **提出单一假设**
   - 清楚陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体明确，不要含糊

2. **最小化测试**
   - 做**最小可能的变更**来验证假设
   - 一次只改一个变量
   - 不要同时修复多个问题

3. **继续前先验证**
   - 有效？→ 进入第四阶段
   - 无效？→ 提出新假设
   - **不要在上面叠加更多修复**

### 第四阶段：实施

**修复根因，而非症状**：

1. **创建失败的测试用例**
   - 最简单的复现
   - 尽可能自动化测试
   - 修复前必须有

2. **实施单一修复**
   - 针对已确认的根因
   - 一次只改一处
   - 不要顺手优化
   - 不要捆绑重构

3. **验证修复**
   - 测试通过了吗？
   - 其他测试没有broken？
   - 问题真的解决了吗？

4. **如果修复无效**
   - 停下来
   - 计算：你已经尝试了几次修复？
   - 少于 3 次：回到第一阶段，用新信息重新分析
   - **3 次或以上：停下来质疑架构（见下方第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **3 次以上修复失败：质疑架构**

   **表明架构有问题的信号**：
   - 每次修复都在不同地方暴露新的共享状态/耦合问题
   - 修复需要"大规模重构"才能实现
   - 每次修复都在其他地方产生新症状

   **停下来质疑根本问题**：跟用户讨论后再继续

## 危险信号 —— 立即停下并遵循流程

如果你发现自己在想：
- "先临时修一下，以后再查原因"
- "试试改 X 看看行不行"
- "改多个地方，跑一下测试"
- "跳过测试，我手动验证"
- "大概是 X 的问题，修一下"
- "我不完全理解，但这可能有用"

**以上所有想法意味着：停下来。回到第一阶段。**

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程处理简单 Bug 很快。 |
| "紧急情况，没时间走流程" | 系统化调试比盲猜瞎试更快。 |
| "先试一下，再调查" | 第一次修复定下基调。从一开始就做对。 |
| "修好后再写测试" | 未经测试的修复不可靠。先写测试来证明。 |
| "同时修多个问题节省时间" | 无法确认哪个有效。还会引入新 Bug。 |
| "再试一次修复"（已失败 2 次以上） | 3 次以上失败 = 架构问题。质疑架构，不要再修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解发生了什么以及为什么 |
| **2. 模式** | 找到正常案例、对比 | 识别差异 |
| **3. 假设** | 提出理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 真实影响

从调试实践中观察到：
- 系统化方法：15-30 分钟修复
- 盲猜方法：2-3 小时瞎折腾
- 一次修复成功率：95% vs 40%
- 引入新 Bug：几乎为零 vs 经常发生

## 交付输出

- 根因假设
- 最小化验证步骤
- 修复前测试或复现方式
- 修复后验证结果
- 若问题跨越共享包装层 / 转换层，附上对应审计结论
- 若连续失败 3 次以上，明确提出架构质疑或升级处理建议

### 示例：测试失败根因定位

- 输入：失败测试、错误日志、最近代码变更
- 输出：根因假设、最小化验证、修复前测试、修复后验证结果
- 约束：在第一阶段未完成前，不得直接给出修复代码
