# 记忆管理器

## 使用时机

- 同类提醒重复出现
- 设计或逻辑决策已经稳定
- 用户偏好已经反复确认
- 事故或回归值得长期记住
- 里程碑或项目复盘时需要归档旧条目

## 必做步骤

1. 只记录稳定事实，不记录短期任务
2. 把条目写入 `.ai-os/memory.md`
3. 分类到：
   - 设计决策
   - 逻辑与契约决策
   - 工程约束
   - 用户偏好
   - 已知坑点
   - 技术债（由 `/postmortem` 审计产生，记录模式漂移、缺失测试、架构违规、过时依赖等）
4. 新条目的活跃度默认为 `active`
5. 更新元数据中的活跃条目数和归档条目数

## 归档策略

当以下条件成立时，将条目从活跃区移到归档区：

- 决策已被新决策明确取代（注明取代条目 ID）
- 工程约束因技术栈变更已失效
- 坑点已被根本修复且不再需要绕行
- `/postmortem` 复盘中确认某条目不再影响当前开发

归档时保留原文，标记 `活跃度: archived`，注明归档原因和归档日期。

## 新 session 恢复优先级

1. 先加载所有 `active` 条目
2. `archived` 条目仅在回溯特定决策历史时按需查阅
3. 如果活跃条目超过 20 条，在 `/resume` 时只加载最近 3 个里程碑内产生的活跃条目，其余标注"存在更早活跃条目，需要时可展开"

## 交付输出

- `.ai-os/memory.md`

## 团队协作

多人并行开发时，`memory.md` 通过 `.gitattributes` 的 `merge=union` 策略减少合并冲突，但仍需注意：

- 新增条目时使用唯一递增 ID（DD-001, CD-002 等），不同开发者追加不同 ID 的条目
- 避免同时修改同一条目的内容；如需修订已有条目，在 PR 中明确标注
- PR 合并时若出现冲突，保留双方条目（决策记录不可丢弃）
- 归档操作（将条目从 active 移至 archived）建议在主干分支上统一执行，避免归档冲突

## 禁止事项

- 禁止把"今天下一步做什么"写进 memory
- 禁止把未确认猜测写成稳定结论
- 禁止直接删除条目；不再有效的条目必须归档而非删除

## 维护信息

- 来源：重复提醒、稳定决策、事故复盘、里程碑归档、技术债审计
- 更新时间：2026-03-26
- 已知限制：本 Skill 不记录短期状态
