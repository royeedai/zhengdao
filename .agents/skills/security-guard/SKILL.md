# 安全审计守卫 (Security Guard)

本 Skill 用于对代码进行结构化的安全漏洞与越权逻辑审查，弥补常规功能测试无法触及的安全盲区。

## 使用时机

- 涉及订单、支付、权限、删除、状态流转等敏感操作
- 用户明确要求做安全审计
- 提交高风险业务代码前需要做静态安全检查

## 使用方式

当被触发时，你需要作为一位资深安全专家，严格遵循以下核查清单对代码进行静态审计。

---

## 核心安全检查清单

### 一、 越权与访问控制 (Access Control & IDOR)
- [ ] **水平越权防范**：当操作某条数据时，是否校验了该数据确实属于当前登录用户？（不能只按数据 ID 查询，必须同时限定用户 ID）
- [ ] **垂直越权防范**：管理后台接口或高权限操作是否**强制**走鉴权中间件，是否在每个处理器或通过切面进行权限检查？
- [ ] **敏感参数防篡改**：价格、运费等敏感金额，是否绝对信任前端传参？（正确做法：前端可传 ID，价格必须后端从数据库重新拉取计算）

### 二、 注入与执行 (Injection)
- [ ] **SQL 注入防范**：所有的数据库交互是否全部使用了参数化查询（ORM 绑定变量 / 预编译语句），绝对禁止字符串拼接查询条件。
- [ ] **XSS 跨站脚本防范**：前端展示用户输入的富文本或备注时，是否进行了转义？若使用了原始 HTML 渲染功能必须格外小心。

### 三、 业务逻辑漏洞 (Business Logic)
- [ ] **并发防抖与幂等性**：对于创建订单、支付这类核心操作，同一时间的高并发请求是否会导致“一单多扣”或“超卖”？（建议：唯一防重 Token、DB 唯一约束、或 Redis 加锁）。
- [ ] **状态机倒退/跨越**：订单等状态流转是否有严格的“限定向”校验？禁止未经鉴权或校验就随意传入任意 status 并保存。
- [ ] **条件竞争与数据覆盖**：是否发生了典型的“先查后写 (Read-Modify-Write)”并发覆盖？（例如：更新积分，应用 `UPDATE points = points + 10 WHERE id=X` 而非 `points=当前取出值+10; db.Save()`）。
- [ ] **乐观锁/版本控制**：编辑操作是否携带版本号或更新时间戳进行乐观锁校验，防止多人同时编辑时后者覆盖前者？

### 四、 基础设施安全
- [ ] **API 限流**：公开接口是否做了频率限制（Rate Limiting），防止恶意高频调用？
- [ ] **文件上传安全**：若支持文件上传，是否校验了文件类型、大小限制，并重命名文件防止路径穿越？

### 五、 数据安全与防泄漏
- [ ] **敏感数据脱敏**：对外暴露的 API 中，手机号、身份证、真实姓名是否实现了掩码输出？
- [ ] **不必要的字段泄露**：API 返回用户资料时，是否移除了密码哈希、私钥等敏感信息？（必须在 DTO/序列化层显式排除）
- [ ] **软删除隔离**：所有带软删除的表在正常业务查询时是否真的过滤了已删除的记录？

## 验收审计报告格式

在完成审计后，向用户输出一份基于安全等级的报告：

```markdown
# 🛡️ 安全审计报告：[模块名称]

## 🚨 P0 - 严重漏洞 (阻塞交付)
- [描述发现的可以导致越权、数据泄露、金额损失的漏洞]
- 修改建议：...

## ⚠️ P1 - 中等风险 (建议修复)
- [描述可能被滥用或存在潜在性能/并发风险的点]
- 修改建议：...

## ✅ 审计通过项
- 已确认金额不可被前端覆盖。
- 已确认手机号全部脱敏。
...
```

## 交付输出

- P0 / P1 风险列表
- 审计通过项
- 修复建议或待验证项
- 是否阻塞交付的结论

## 约束

- 不要把普通代码质量问题伪装成安全问题
- 不要在没有证据时宣称存在漏洞
- 本 Skill 负责静态安全审计，不替代渗透测试、上线审批或生产监控

### 示例：订单状态流转安全检查

- 输入：订单接口、状态流转代码、权限判断逻辑
- 输出：P0 / P1 风险列表、通过项、修复建议
- 约束：如果无法证明存在越权链路，应标记为待验证而不是直接定性
