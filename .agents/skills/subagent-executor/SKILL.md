# 子代理驱动执行器

## 核心原则

```
每任务一个隔离子代理 + 两阶段审查 = 高质量、快迭代
```

子代理拥有独立上下文，不继承主控 agent 的会话历史。主控 agent 负责精确构造子代理所需的全部信息，保证子代理专注完成任务，同时保护主控自身上下文用于协调。

## 使用时机

- `/build` 阶段，`tasks.yaml` 已就绪
- 任务大部分可独立执行
- 希望在同一会话中连续推进多个任务

不适用场景：
- 任务高度耦合、必须共享大量上下文状态
- 尚未完成设计门和逻辑门
- 只有 1-2 个极简任务，直接执行更高效

## 执行流程

### 第一步：准备

1. 读取 `tasks.yaml`，提取所有待执行任务的完整文本、context_files 和 derived_checks
2. 读取 `MISSION.md`、`DESIGN.md`、相关 spec，准备场景上下文
3. 用 TodoWrite 创建任务追踪列表

**关键：提前提取全部任务内容。子代理不读 plan 文件，主控提供全文。**

### 第二步：逐任务执行

对每个任务重复以下循环：

#### 2a. 分配实现子代理

根据任务复杂度选择模型：

| 信号 | 模型选择 |
|------|---------|
| 改动 1-2 个文件，spec 完整明确 | 快速模型 |
| 跨多文件、有集成关注点 | 标准模型 |
| 需要设计判断或广泛代码库理解 | 最强模型 |

构造 prompt 时必须提供：
- 任务完整描述（不是摘要）
- 相关文件路径和当前内容摘要
- 项目约束和设计契约
- 目标模块的已声明依赖清单和模块边界，明确禁止引入循环依赖或未在构建配置中声明的依赖
- TDD 要求：先写测试、看失败、再实现
- 明确的交付标准

详见 `references/implementer-prompt.md`。

#### 2b. 处理子代理状态

子代理返回四种状态之一：

| 状态 | 处理方式 |
|------|---------|
| `DONE` | 进入 spec 审查 |
| `DONE_WITH_CONCERNS` | 先读取顾虑，若涉及正确性或范围则先解决，若只是观察则记录后进入审查 |
| `NEEDS_CONTEXT` | 提供缺失上下文，重新分配同一任务 |
| `BLOCKED` | 评估阻塞原因（见下方阻塞处理） |

**阻塞处理协议**：
1. 上下文不足 → 补充上下文，重新分配，使用同一模型
2. 任务需要更强推理 → 使用更高能力模型重新分配
3. 任务过大 → 拆成更小的子任务
4. 计划本身有问题 → 暂停，报告用户

**绝对禁止**忽略子代理的升级信号或强制同一模型重试而不做任何变更。

#### 2c. 第一阶段审查：Spec 一致性

分配 spec 审查子代理，检查：
- 是否实现了 spec 中要求的所有内容
- 是否添加了 spec 未要求的内容
- 命名、状态枚举、字段映射是否与设计契约一致

审查结论：
- 通过 → 进入代码质量审查
- 未通过 → 实现子代理修复问题 → 重新审查

详见 `references/spec-reviewer-prompt.md`。

#### 2d. 第二阶段审查：代码质量

分配代码质量审查子代理，检查：
- 测试覆盖和测试质量
- 代码清晰度和可维护性
- 错误处理和边界情况
- 是否遵循项目既有模式
- 依赖链完整性：新增/删除的 import 是否与构建配置中声明的依赖一致，是否引入了循环依赖

审查结论：
- 通过 → 标记任务完成
- 未通过 → 实现子代理修复问题 → 重新审查

**严格顺序：必须先通过 spec 审查，再进入代码质量审查。**

详见 `references/quality-reviewer-prompt.md`。

#### 2e. 标记完成

更新 TodoWrite 和 `tasks.yaml` 中的任务状态。

### 第三步：全量终审

所有任务完成后，分配一个终审子代理对整体实现做最终审查：
- 所有需求是否覆盖
- 组件间集成是否正确
- 是否存在跨任务引入的不一致
- 跨任务依赖链合并后的整体一致性：是否有任务删除了其他任务仍在使用的依赖，是否引入了跨模块循环依赖
- 共享文件完整性：路由注册、入口文件、barrel export、全局配置等在多任务修改后是否完整（无定义丢失或重复）

### 第四步：收尾

- 更新 `STATE.md`
- 进入 `/verify` 或按项目约定处理分支

## 并行调度（可选）

当存在 3 个以上互相独立且无共享状态的任务时，可以并行分配多个实现子代理：

**使用条件**：
- 任务改动不同文件
- 无共享状态依赖
- 不会产生 git 冲突

**禁止并行的情况**：
- 任务间存在数据依赖
- 改动同一文件或同一模块
- 需要顺序验证

**共享文件处理**：若多个并行任务都需要修改同一共享文件（路由注册、入口文件、barrel export、全局配置等），该文件的修改必须串行执行，或在并行完成后做合并完整性校验，确保不会出现定义丢失或重复。

并行完成后仍需逐个进行两阶段审查。

## 与 AI-OS 治理的集成

- 命中 `approval_required` 的任务：在子代理执行前暂停等待用户确认
- 高风险任务：使用最强模型 + 强制两阶段审查 + 终审前暂停
- 子代理执行中发现问题超出任务边界：立即暂停，报告主控，由主控决定是否升级到 `/change-request`
- 所有审查结论和证据必须回写 `tasks.yaml` 和 `STATE.md`

## 禁止事项

**绝对禁止**：
- 在 main/master 分支上直接开始实现（未经用户同意）
- 跳过任一审查阶段
- 带着未修复的审查问题进入下一任务
- 并行分配会产生冲突的实现子代理
- 让子代理自行读取 plan 文件（必须由主控提供全文）
- 忽略子代理的问题或顾虑
- 在 spec 审查未通过时开始代码质量审查
- 子代理失败时尝试手动修复（应分配修复子代理）

## 交付输出

- 每个任务的实现代码和测试
- 每个任务的 spec 审查结论
- 每个任务的代码质量审查结论
- 全量终审结论
- 更新后的 `tasks.yaml` 和 `STATE.md`
