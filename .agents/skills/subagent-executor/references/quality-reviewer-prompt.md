# 代码质量审查子代理 Prompt 模板

前置条件：该任务已通过 spec 一致性审查。

## 模板结构

```markdown
# 代码质量审查

## 项目约定
[项目使用的技术栈、代码风格、测试框架、既有模式]

## 实现变更
[git diff 或变更文件的完整内容]

## 审查要点

### 测试质量
- 是否遵循 TDD（先测试后实现）？
- 测试是否覆盖正常路径和异常路径？
- 测试是否测试真实行为（而非 mock 行为）？
- 测试命名是否清晰描述被测行为？

### 代码质量
- 代码是否清晰可读？
- 是否遵循项目既有模式和约定？
- 错误处理是否完善？
- 是否有明显的性能问题？

### 工程标准
- 是否有未使用的导入或变量？
- 类型定义是否完整（TypeScript 项目）？
- 是否存在硬编码的魔术数字或字符串？

## 交付格式

- 结论: APPROVED / NEEDS_CHANGES
- 优点: [做得好的地方]
- （如 NEEDS_CHANGES）问题清单，按严重程度分级:
  - Critical: [必须修复，阻塞通过]
  - Important: [应该修复]
  - Suggestion: [建议改进，不阻塞]
```

## 注意事项

- 只有在 spec 审查通过后才进行代码质量审查
- Critical 问题必须修复后重新审查
- Important 问题应修复，除非有合理说明
- Suggestion 记录但不阻塞
