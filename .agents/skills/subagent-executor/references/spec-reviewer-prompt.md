# Spec 一致性审查子代理 Prompt 模板

## 模板结构

```markdown
# Spec 一致性审查

## 任务 Spec
[从 spec / tasks.yaml 中提取的该任务需求原文]

## 设计契约
[从 DESIGN.md 中提取的相关接口、命名、状态枚举约定]

## 实现变更
[git diff 或变更文件列表和关键变更内容]

## 审查要点

逐项检查：

1. **完整性**：spec 要求的每一项是否都已实现？列出每个需求点和对应实现
2. **无多余实现**：是否存在 spec 未要求的功能？（YAGNI 原则）
3. **契约一致性**：命名、字段、状态枚举、接口签名是否与设计契约一致？
4. **边界处理**：spec 中提到的异常路径、空数据、权限检查是否覆盖？

## 交付格式

- 结论: PASS / FAIL
- （如 FAIL）问题清单:
  - 缺失: [spec 要求但未实现的内容]
  - 多余: [spec 未要求但被添加的内容]
  - 偏差: [与设计契约不一致的地方]
```

## 注意事项

- 审查的是 spec 一致性，不是代码质量（代码质量由下一阶段审查）
- 不接受"差不多""基本一致"——必须精确对照
- 发现问题后由原实现子代理修复，修复后重新审查
