# API 级检查清单（详细版）

本文件是 `fullstack-dev-checklist` Skill 的补充参考，提供 API 开发的详细检查规范。

---

## RESTful API 设计规范

### URL 设计
```
GET    /api/v1/products          # 列表查询
GET    /api/v1/products/:id      # 详情查询
POST   /api/v1/products          # 创建
PUT    /api/v1/products/:id      # 全量更新
PATCH  /api/v1/products/:id      # 部分更新
DELETE /api/v1/products/:id      # 删除
```

### 统一响应格式
```json
// 成功
{
  "code": 0,
  "message": "success",
  "data": { ... }
}

// 列表成功
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [...],
    "total": 100,
    "page": 1,
    "page_size": 20
  }
}

// 失败
{
  "code": 40001,
  "message": "商品名称不能为空",
  "data": null
}
```

### 错误码设计
| HTTP 状态码 | 业务错误码 | 说明 |
|------------|-----------|------|
| 400 | 40001-40099 | 参数校验失败 |
| 401 | 40100 | 未登录 |
| 403 | 40300 | 无权限 |
| 404 | 40400 | 资源不存在 |
| 409 | 40900 | 资源冲突（如名称重复） |
| 500 | 50000 | 服务器内部错误 |

## 每个 API 接口的必查项

### 1. 列表查询接口
- [ ] 支持分页参数：`page` (默认1), `page_size` (默认20, 最大100)
- [ ] 支持排序参数：`sort_by`, `sort_order` (asc/desc)
- [ ] 支持筛选参数：与业务相关的筛选条件
- [ ] 支持关键字搜索：`keyword` 参数
- [ ] 返回数据包含 `total` 总数
- [ ] 关联数据预加载（商品列表包含分类名称、创建者名称等）
- [ ] 软删除的数据不返回
- [ ] 默认排序合理（通常按创建时间倒序）

### 2. 详情查询接口
- [ ] 路径参数 ID 校验（格式、存在性）
- [ ] 关联数据完整返回
- [ ] 权限校验（用户只能查看自己有权限的数据）

### 3. 创建接口
- [ ] 所有必填字段校验
- [ ] 字段格式校验（手机号、邮箱等）
- [ ] 字段长度校验
- [ ] 唯一性校验（如名称不能重复）
- [ ] 关联数据存在性校验（分类ID是否有效）
- [ ] 创建成功返回完整对象（含生成的ID）
- [ ] 记录创建人和创建时间

### 4. 更新接口
- [ ] ID 存在性校验
- [ ] 所有校验规则同创建接口
- [ ] 更新唯一字段时排除自身（如改名时检查是否与其他记录重名）
- [ ] 更新成功返回更新后的完整对象
- [ ] 记录更新时间
- [ ] 处理并发更新（乐观锁或版本号）

### 5. 删除接口
- [ ] ID 存在性校验
- [ ] 关联数据检查（有子数据时是否允许删除，或提示用户）
- [ ] 优先使用软删除
- [ ] 返回删除确认

## Go 后端代码分层规范

```
handler/   → 接收请求、参数绑定、调用 service、返回响应
service/   → 业务逻辑（不直接操作数据库）
repository/→ 数据库操作（CRUD）
model/     → 数据库模型定义
dto/       → 请求/响应数据结构定义
middleware/→ 中间件（认证、日志、错误处理）
```

每层的职责：
- **Handler**：只做参数绑定 + 调用 Service + 格式化响应，不写业务逻辑
- **Service**：处理业务规则、数据转换，调用一个或多个 Repository
- **Repository**：纯数据库操作，不含业务逻辑
- **Model**：定义数据库表结构、字段标签
- **DTO**：定义 API 入参/出参结构，包含验证标签
