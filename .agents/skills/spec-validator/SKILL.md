# Spec 校验器

## 使用时机

- 生成或更新 spec
- 准备从 `/design` 进入 `/plan`
- 准备从 `/plan` 进入 `/build`

## 使用方式

对目标 spec 逐项执行下方「校验重点」清单，输出通过项和缺口清单。可在 `/design` 结束时、`/plan` 开始前和 `/build` 开始前各执行一次。

## 校验重点

- 是否能回答“这个模块到底要做什么”
- 是否先判断并写清交互模式：`sync` / `streaming` / `async-job` / `event-driven`
- 若存在增量输出、长耗时、异步完成或持续反馈，是否明确说明为什么选择当前模式，以及为什么不用其他模式
- 是否写清关键流程和状态流转
- 是否写清关键输入、输出、字段和异常路径
- 是否写清契约基准、字段映射 / 适配说明和集成触点
- 是否区分了必须先做对的逻辑与允许后补的细节
- 是否写清正常路径和异常 / 空数据路径分别需要什么证据才能验收

## 交付输出

- 可进入 `/plan` 的 spec
- 或需要补齐的缺口清单

### 示例：在进入 build 前校验协议模式

- 输入：包含聊天、长耗时处理或事件回调的模块 spec
- 输出：补齐后的 `交互模式`、`契约基准`、`异常/空数据证据` 缺口清单

## 禁止事项

- 禁止只有页面描述，没有逻辑规则
- 禁止把天然流式 / 异步场景静默默认成普通 request / response
- 禁止只有接口字段，没有状态流转
- 禁止没有契约基准和字段映射说明就进入跨层实现
- 禁止 spec 无法支撑验收门
