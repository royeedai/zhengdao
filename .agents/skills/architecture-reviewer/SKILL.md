# 代码架构与内聚审查 (Architecture & Clean Code Review)

> 本 Skill 适用于任何技术栈（Go/Java/Python/Node/Rust 等）。

本 Skill 不关注"功能是否实现"，而专注审查"代码写得好不好、够不够优雅、是否利于后期扩展"。

## 使用时机

- 大中型重构后需要做架构复查
- 完成复杂功能后，需要评估代码质量和可维护性
- `code-review-guard` 识别到存在明显架构异味

## 使用方式

作为一名架构师，参照以下原则审查目标代码结构。

---

## 审查维度与 Clean Code 原则

### 一、 单一职责原则 (SRP)
- [ ] **上帝类/函数**：是否存在动辄数百行的 Handler 或 Service 承担了过多的指责（例如一个函数里既查 DB，又发 MQ，又组装巨型复杂对象）？
- [ ] **逻辑越界**：Handler 层是否偷写了本该属于 Service 层的业务逻辑？Repository 层是否夹杂了鉴权逻辑？（标准的 Controller - Service - Repository 必须泾渭分明）。

### 二、 DRY (Don't Repeat Yourself) 防重复
- [ ] **复制粘贴坏味道**：是否存在两处以上的代码在做几乎一模一样的数据转换、校验或异常捕获？是否应该提炼成公共函数或中间件？
- [ ] **跨文件重复声明 [P0/严重]**：同一个 package/目录中，是否存在多个文件定义了同名的 struct/class/function/interface？如果两个文件都定义了 `CategoryRepository`，则必须合并到唯一处，否则编译将直接失败。
- [ ] **Magic String / Magic Number**：代码里是否四处散落着硬编码的 `status == 2` 或 `"role_admin"`？是否应该统一抽取并定义为具有业务语义的常量/枚举？

### 三、 错误与异常处理
- [ ] **不吞噬异常**：是否所有返回的错误都被正确捕获和处理/抛出，而不是简单地忽略？
- [ ] **有效的信息追踪**：错误返回给上层时，是否携带了足够的上下文以便日后 Debug，而不是直接返回笼统的服务器错误？
- [ ] **错误字典归一**：业务错误（如商品不存在、名称重复）是否定义了统一的 Error 变量表，前端能否依赖统一的 Error Code 做出可预测的交互？

### 四、 防护性编程与健壮性
- [ ] **空指针恐慌防护**：在针对结构体嵌套对象或数组进行取值、遍历前，是否做了边界与空值拦截验证？
- [ ] **默认值/零值陷阱**：在结构体序列化/更新 DB 字段时，对零值的处理是否正确？（例如更新 `0`、`false`、`""` 时，是否误用了非指针/非可空类型，导致零值无法更新入库？）

### 五、 “代码可读性” 即 “文档”
- [ ] **命名达意**：变量名和函数名是否不言自明？拒绝 `a`, `b`, `flag1`，提倡 `hasUnfinishedOrders`, `totalDiscountAmount`。
- [ ] **注释的双向溯源**：**[强制]** 核心的业务约束、复杂的表单逻辑处，是否留有对应的 `// 对应 .spec: FR-XXX` 注释，确立代码与需求文档的追踪锚点？

### 六、 增量开发卫生 (Incremental Development Hygiene)
> 基础检查项（共享模型补全、入口同步、类型正确性）请参照 `fullstack-dev-checklist` 维度零。本处仅审查架构层面的增量卫生：
- [ ] **新模块是否遵循了已有的分层约定**：是否沿用了项目已有的 Controller-Service-Repository 分层？还是另起炉灶自成一派？
- [ ] **跨模块依赖方向是否合理**：新模块是否引入了反向依赖（下层模块反向依赖上层模块）？

## 约束

- 不要把功能是否实现混进架构审查结论
- 不要为了追求抽象而鼓励过度设计
- 本 Skill 负责可维护性和抽象质量，不替代功能验收与安全审计

## 评审输出模板

```markdown
# 🏛️ 架构与 Clean Code 评审报告

## ❌ 架构异味 (亟需重构，P0/P1)
1. **[违反单一职责]**：文件 `XXX` 的 `YYY` 函数过于臃肿。
   - 重构建议：将这部分逻辑抽离为 `ZZZ` 组件。
2. **[硬编码问题]**：...

## 🔧 代码级坏味道 (建议优化，P2)
1. **未处理零值陷阱**：发现多处...
2. **缺乏上下文记录**：...

## 💡 闪光点
- 发现使用了工厂模式/策略模式极大地解耦了结构...
- Service 和 Repository 分层非常清晰...
```

### 示例：识别上帝类

- 输入：单个超长 Service / Handler 文件
- 输出：架构异味说明、重构方向、优先级
- 约束：建议必须落到具体职责拆分，而不是空泛说“应该优化”
