# TDD 自我合理化防御

当你发现自己在想以下任何一条时，停下来，这是在合理化跳过 TDD。

## 合理化对照表

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会出错。写测试只要 30 秒。 |
| "我先实现，后补测试" | 后补的测试立即通过，什么也证明不了。 |
| "后补测试也能达到同样效果" | 后补测试回答"代码做了什么"；先写测试回答"代码应该做什么"。 |
| "已经手动测试过了" | 手动测试是临时的，没有记录，无法重跑。 |
| "删掉 X 小时的代码太浪费" | 沉没成本谬误。保留不可信的代码才是真正的浪费。 |
| "保留当参考，然后先写测试" | 你会"适配"它。这就是后补测试。删掉就是删掉。 |
| "需要先探索一下" | 可以探索，但探索完后删掉，用 TDD 重做。 |
| "测试太难写说明..." | 说明设计太复杂。听从测试的信号，简化接口。 |
| "TDD 会让我变慢" | TDD 比事后调试更快。务实 = 先写测试。 |
| "手动测试更快" | 手动测试不能证明边界情况。每次改动都要重测。 |
| "现有代码没有测试" | 你在改进它。为你接触的代码补上测试。 |
| "TDD 太教条了，我更务实" | TDD 就是务实：发现 bug 在提交前、防止回归、记录行为、支持重构。 |

## 危险信号 — 停下，从头开始

出现以下任何情况，必须停下来，删掉代码，从 TDD 开始：

- 先写了代码再写测试
- 实现完成后才补测试
- 测试第一次就通过了（没有看到失败）
- 无法解释测试为什么失败
- 测试是"以后加"的
- 在合理化"就这一次"
- 在说"我已经手动测试了"
- 在说"后补测试也一样"
- 在说"保留当参考"或"适配已有代码"
- 在说"已经花了 X 小时，删掉太浪费"
- 在说"TDD 太教条，我更务实"
- 在说"这个不一样因为..."

**以上所有意味着：删掉代码，用 TDD 从头开始。**
