# 派生规则参考

这些规则不是新的高层原则，而是对 AI-OS 五条核心要求的具体落地。

## 1. 目标与用户确认优先

### 1.1 先做局部判断，再决定是否扩散

- 用户明确点名模块、接口、字段、页面或命令时，先做局部判断
- 只有当位置不清、边界不清、现有模式不清时，才扩散到 brownfield 结构梳理

### 1.2 质量优先级必须写清

- 需要明确：逻辑正确、关键体验、工程可维护、交付速度的先后次序
- 若用户未明确，默认：逻辑正确 > 关键体验 > 工程可维护 > 交付速度

### 1.3 关键选型必须确认

- 当技术栈、目标运行态、关键集成方式或不可逆工程方案会影响交付质量时，必须显式确认
- 允许保留待确认项，但不得在 build 前静默替用户拍板

### 1.4 "配置 / 设置 / 选项"必须轻量确认操作闭环

- 需求里出现"可设置""可配置""可维护"等表述时，先确认它是静态预置、后台可配，还是需要用户 / 运营入口
- 这是轻量追问，不等于一刀切要求 UI CRUD

## 2. 关键设计与逻辑先锁定

### 2.1 先锁关键页面与关键流程

- 没有关键页面、信息架构和关键流程定义时，不进入完整开发
- 允许先锁关键页面，再逐步扩展细节

### 2.2 reverse-spec 必须保留对照边界

- 参考对象只是来源，不是自动兼容目标
- 必须记录哪些行为必须一致、哪些允许改写
- 必须保留关键页面 / API / 流程的对照证据来源

### 2.3 brownfield 必须先锁共享基础设施基准

- request wrapper / interceptor、DTO / adapter、中间件、路由鉴权和全局样式变量等共享基础设施，会改变局部实现的真实契约
- 这些约定未审计前，不要把局部页面 / 接口文件的表层结构直接当最终基准

## 3. 自适应治理

### 3.1 先判断项目模式

- `greenfield`：从想法起步
- `reverse-spec`：有截图 / API / 参考源码
- `brownfield`：已有仓库待接手
- `change`：已有项目上的变更

### 3.2 工件深度按风险缩放

- 探索档：Mission + State + 最小设计笔记
- 标准档：再加 Design + Spec + Tasks + Acceptance
- 高风险档：再加 Risk + Release + 目标运行态证据
- 命中资产、权限、不可逆状态流转、跨用户数据操作、并发敏感更新或外部副作用时，必须硬升级到高风险档

### 3.3 执行力阶梯指导

- 项目中反复违反的约定，应从文档（L2）提升到模板（L3）或自动化检查（L4）
- 关键门禁（如回归检测、格式化、类型检查）推荐配置为 CI / pre-commit hook（L5）
- `.ai-os/CONVENTIONS.md` 是 L2 级约定，项目成熟后应逐步提升到 linter 规则（L4）或 CI 检查（L5）
- `/postmortem` 应评估哪些经验教训需要提升执行力级别

## 4. 证据化完成

### 4.1 四门必过

- 设计确认门
- 逻辑确认门
- 实现质量门
- 交付质量门

### 4.2 reverse-spec 额外检查 parity-gate

- 必须至少对关键页面、关键 API 或关键流程提供一种对照证据

### 4.3 不接受伪完成

- 只有入口但无法完成关键任务，不算完成
- 未实现、未验证、仅占位或仅 demo 的能力，不算完成
- 只有正常路径成立、异常 / 空值 / 权限拒绝 / 超时路径一触即碎，不算完成

### 4.4 契约基准先于实现

- spec 必须显式记录交互模式、契约基准、字段映射 / 适配说明和集成触点
- build / verify 必须对照契约基准检查跨层字段、状态枚举和错误语义是否一致

### 4.5 交付必须显式暴露静态校验与人工动作

- 至少保留一项项目原生静态校验证据：compile / type-check / build 其一
- 交付说明必须区分 `AI 已完成` 与 `需人工执行`
- 仍需人工执行的 SQL / 重启 / 迁移 / 补数 / 环境变更，不能写成"已全部完成"

### 4.6 回归零容忍

- `/build` 每个 wave 完成后，原本通过的测试不允许变为失败；如果有测试套件，必须执行回归基线对比
- `/verify` 存在回归时不能放行，必须先修复回归再继续
- 回归对比结论（基线测试数 / 当前通过数 / 回归数）是交付证据的一部分

### 4.7 最小改动原则

- 每个 wave 的实际改动文件应在 wave 计划的 `context_files` 范围内
- 超出范围的改动必须暂停说明原因，而不是静默扩散
- 禁止在一个 wave 中做"顺手优化"不相关文件

## 5. 可恢复的项目记忆

### 5.1 STATE 优先

- 新 session 先看 `STATE.md`
- 最小阅读集优先由 `STATE.md` 决定

### 5.2 memory 只记稳定事实

- 设计决策
- 逻辑 / 契约决策
- 工程约束
- 用户偏好
- 已知坑点
