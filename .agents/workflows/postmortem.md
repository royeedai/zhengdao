# /postmortem

当项目交付、里程碑结束、重大问题关闭或需要系统复盘时触发。

## 目标

完成完整复盘，并把稳定结论沉淀进共享 `.ai-os/memory.md`；若稳定代码模式发生变化，同步更新 `.ai-os/CONVENTIONS.md`。

## 必做步骤

1. 读取 `MISSION.md`、`DESIGN.md`、`tasks.yaml`、`acceptance.yaml`、`STATE.md`、交付证据与相关问题记录
2. 引导复盘至少覆盖：
   - 交付结果
   - 问题根因
   - 踩坑记录
   - 优化方向
   - 规则 / 流程迭代建议
3. **技术债审计**：评估本轮是否引入了新的技术债（模式漂移、缺失测试、架构违规、过时依赖），是否消除了旧的技术债，趋势是上升还是下降；将技术债结论同步到 `memory.md` 的技术债追踪区
4. **CONVENTIONS.md 审查**：检查共享 `.ai-os/CONVENTIONS.md`（如存在）是否仍然反映项目实际的代码模式；如果项目中出现了新的稳定模式或旧约定已过时，更新约定
5. **执行力评估**：评估本轮中哪些经验教训需要提升执行力级别（从文档提升到模板、自动化检查或 CI 钩子）
6. 区分“稳定结论”和“临时观察”
7. 将稳定结论同步更新到共享 `.ai-os/memory.md`
8. 若本轮形成了新的稳定代码模式、反模式或跨 lane 通用约束，同步更新共享 `.ai-os/CONVENTIONS.md`
9. 若当前 lane 准备关闭，明确 `memory.md` / `CONVENTIONS.md` 的回流结果是 `done` 还是 `not-needed`，避免 lane 归档后仍把稳定经验留在私有工件里
10. 审查 `memory.md` 中的现有活跃条目，将已被取代、已失效或不再影响当前开发的条目归档到归档区（参照 `memory-manager` skill 的归档策略）
11. 在 `STATE.md` 中记录本轮复盘已完成及后续跟进项

## 输出

- 复盘摘要
- 需进入 `memory.md` 的稳定经验
- 需进入 `CONVENTIONS.md` 的稳定模式或反模式（如有）
- 技术债变化结论（新增 / 消除 / 趋势）
- CONVENTIONS.md 更新建议（如有）
- lane 收口回流结论（如当前 lane 准备归档）
- 执行力提升建议（哪些规则应从文档升级到自动化检查）
- 归档条目清单（如有）
- 后续优化建议

## 禁止事项

- 禁止只写结果，不分析根因
- 禁止把未经确认的猜测沉淀进 `memory.md`
- 禁止复盘完成后不更新项目记忆
- 禁止 lane 已准备归档，却不先判断共享 `memory.md` / `CONVENTIONS.md` 是否需要回流
- 禁止直接删除 memory 条目；不再有效的条目必须归档而非删除
