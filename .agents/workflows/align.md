# /align

当用户提出一个新项目、模糊想法、复刻目标、老项目需求变更，或任何"先把方向说清"的任务时触发。

> 路径说明：lane 布局下，下文提到的 `MISSION.md`、`baseline-log/`、`STATE.md` 默认位于当前 lane 的 `.ai-os/lanes/<lane-id>/`；legacy 单交付项目仍位于 `.ai-os/` 根层。

## 目标

在开始设计和编码前，把以下内容说清并锁进当前 lane 的 `MISSION.md` / `baseline-log/`：

- 宿主项目必要上下文（仅 brownfield / change 需要）
- 当前交付目标
- 用户场景
- 范围边界（做什么 / 不做什么）
- 核心需求
- 非功能性约束
- 成功标准
- 当前确认基线
- 技术选型约束
- 现在属于哪种项目模式：`greenfield` / `reverse-spec` / `brownfield` / `change`
- 当前任务属于 `P0` / `P1` / `P2` 哪一档，以及推荐采用什么治理强度

## 必做步骤

1. 先判断这次工作是继续当前 lane，还是需要新建并行 lane：
   - 若当前 lane 已承载这轮交付目标，继续在当前 lane 对齐
   - 若这是新的并行交付主题、独立 release train，或不应与当前 Mission / Tasks / Acceptance 共用基线，先执行 `create-ai-os lane add <lane-id> .`
   - 若存在多个 active lane 且当前 lane 不明确，先执行 `create-ai-os lane list .`，必要时再用 `create-ai-os lane activate <lane-id> . --only`
2. 识别当前任务是项目级、模块级，还是局部变更
3. 判断当前任务治理档位：
   - 新项目 / 新模块 / 大范围变更：`P0`
   - 已有模块功能新增或非核心调整：`P1`
   - 单点 bug、文案、样式、配置微调：`P2`
4. 判断项目模式：
   - 从想法起步：`greenfield`
   - 参考截图 / API / 源码：`reverse-spec`
   - 先接手现有仓库：`brownfield`
   - 已有项目上的变更：`change`
5. 把用户提供的截图、接口文档、口头描述、现有代码、历史约束整理为结构化需求框架，并强制覆盖 8 个核心模块：
   - 当前交付目标
   - 用户场景
   - 范围边界（做什么 / 不做什么）
   - 核心需求
   - 非功能性约束
   - 成功标准
   - 验收基线
   - 技术选型约束
6. 若是 `brownfield` / `change`，先最小化整理宿主项目上下文：当前涉及的模块、共享基础设施、既有约束和影响边界；不要把整个存量项目重新当成当前 mission
7. 为核心需求建立可追溯的需求点标识，确保后续可拆任务、可做验收、可留验证结论
8. 明确"当前最小可行闭环"和"可后续补充项"
9. 明确质量优先级：逻辑正确、关键体验、工程可维护、交付速度的先后次序
10. 明确关键技术栈、目标运行态、关键工程选型和不可逆方案，区分已确认与待确认
11. 对已确认的关键依赖（框架、运行时、中间件、SDK、基础镜像等）执行实际可用性验证，禁止仅凭 AI 训练数据或搜索结果认定版本存在；验证方式因生态而异（包管理器查询、registry API、manifest 检查等），但结论必须有证据；若生成了构建配置草稿，应执行依赖解析预检（dry-run）确认无冲突
12. 主动识别素材中的模糊点、歧义点、默认假设和待确认边界，整理成问题清单，禁止脑补填空
13. 对出现"配置 / 设置 / 选项"的需求，轻量追问一次操作闭环：它是静态预置、后台可配，还是需要用户 / 运营入口
14. 生成或更新 `MISSION.md`，只保留低频、已确认、共享的当前 lane 交付基线；若是 `brownfield` / `change`，明确区分“宿主项目上下文”和“当前 lane 的交付基准”，避免把整个存量项目重新写成当前 mission
15. 生成或更新 `baseline-log/`，至少新增一条 `BL-YYYYMMDD-HHMMSS-slug.md` 记录，标明当前 lane 基线摘要、影响工件和确认时间
16. 生成或更新 `STATE.md`，记录当前阶段、当前目标、待确认项、确认停点和最小阅读集；待确认项只放在 `STATE.md`，不要写回 `MISSION.md`
17. 向用户输出 `MISSION.md` 核心摘要、`baseline-log/` 最新记录、当前治理档位、待确认项和下一步建议，等待用户明确确认后再离开 `/align`

## 输出

- `MISSION.md`
- `baseline-log/`
- `STATE.md`
- 向用户同步的需求摘要、问题清单和推荐路径：继续 `/design`、进入 `/change-request`，还是先补素材 / 先做 brownfield 结构梳理

## 反模式：这个太简单不需要对齐

不存在"太简单不需要对齐"的任务。简单任务恰恰是未经审视的假设导致最多返工的地方。

对齐的深度可以很轻（P2 可能只需几句话的确认），但必须明确 present 给用户并获得确认：
- P0：完整走 `/align` 所有步骤
- P1：至少确认目标、范围边界和验收标准
- P2：至少用一句话确认"修什么、改到什么程度、怎么验证"

**"看起来很简单"不是跳过对齐的理由。越简单越容易因为没说清就动手而浪费时间。**

## 禁止事项

- 禁止在目标、用户和成功标准不清时直接进入大规模实现
- 禁止把"用户一句话需求"自动脑补成完整产品定义
- 禁止在关键技术选型未确认时由 AI 默认拍板
- 禁止仅凭 AI 训练数据或搜索结果认定依赖版本可用，必须通过实际查询 registry 验证
- 对明确的局部改动，禁止默认先全仓分析
- 禁止在已有 lane 已承载另一条并行交付时，继续把第二套目标硬写进当前 lane
- 禁止未等待用户确认就直接推进到 `/design`、`/plan` 或 `/build`
- 禁止以"太简单"为由完全跳过对齐确认
