# /debug

当用户要求修复单一 bug、微调样式 / 文案 / 配置，或定位一个明确问题时触发。

## 目标

用最小但完整的流程完成"先定方案再动手"，避免越界修复和改 A 坏 B。

## 反模式：这个 Bug 太简单不需要方案确认

即使是最简单的 Bug，也需要最低限度的方案确认。"显而易见"的修复方案常常是错的——它可能修的是症状而不是根因，或者引入的改动超出了预期范围。

最小确认只需一句话：**"问题是 X，根因是 Y，我只改 Z 文件的这个位置，验证方式是 W"**。

如果连这一句话都说不出来，说明你还没有充分理解问题。

## 必做步骤

1. 读取 `.ai-os/STATE.md`、相关 `MISSION.md` / spec / 任务上下文，确认本次问题没有超出 `P2` 边界
2. 明确：
   - 复现路径或现象
   - 当前假设或已确认根因
   - 影响范围
   - 计划修改的文件和模块
   - 是否触及核心逻辑、已确认设计或关键依赖
3. 在首个写操作前先完成只读 tracing / inspect；优先用日志、调用链、共享包装层和现有测试定位问题，禁止通过连续打补丁试错来找根因
4. 当现象跨越组件边界、共享包装层 / 转换层、通用抽象或全局样式基准时，先优先排查：
   - request wrapper / axios interceptor / 统一响应解包
   - DTO / adapter / mapper
   - Router Guard / 鉴权 / 导航骨架
   - 多租户 / 上下文注入
   - BaseEntity / 自动填充 / 通用审计字段
5. 若问题涉及 wrapper / route / schema / 通用抽象复用，先核对真实 schema / route / wrapper 契约，并找一个同仓正常实现对照；若当前实现偏离既有模式，先说明理由
6. 向用户输出修复方案、边界说明、风险和验证计划，禁止未定方案直接改代码
7. 等待用户明确回复"确认修复方案，可执行"
8. 只修改直接相关代码；若执行中发现问题超出原边界，立即暂停并升级到 `/change-request` 或 `/design`
9. 修复时遵循 TDD：先写一个复现该 Bug 的失败测试，确认测试因 Bug 而失败，再实施修复，确认测试通过
10. 若修复触及共享层、跨层契约或通用抽象，每完成一个修复 slice 就先跑一次最小验证，不要等到所有改动都堆完再一起看结果
11. 完成后验证：
   - 目标问题已解决
   - 影响范围内相关功能已回归
   - 若仓库存在 compile / type-check / build 入口，至少执行并记录一项项目原生静态校验
   - 没有引入新的已知回归
12. 若本轮问题已经暴露出稳定 failure mode 或高频回归入口，把最小复现、放行条件和验证方法同步到 `.ai-os/evals/`、`.ai-os/verification-matrix.yaml` 或等价工件
13. 输出完整的验证结果与回归报告，并显式拆成：
   - 代码状态：仓库代码是否已修复
   - 数据状态：是否仍需补 SQL、修种子、修历史记录
   - 运行状态：是否仍需重启服务、刷新浏览器、重新登录、清缓存
14. 更新 `.ai-os/STATE.md`，必要时同步 spec / acceptance 中的验证记录

## 输出

- 修复方案与边界说明
- 修改后的验证和回归结论
- 代码状态 / 数据状态 / 运行状态三分诊断
- 是否需要升级为更高治理档位

## 禁止事项

- 禁止在未给出根因 / 假设、影响范围和文件边界前直接改代码
- 禁止在只读 tracing 未完成前，通过连续打补丁试错定位根因
- 禁止跳过 request wrapper / interceptor / adapter / router guard / BaseEntity 这类共享层排查，就直接在局部业务代码补丁式修复
- 禁止已有同仓正常实现却不对照，另造一套“看似合理”的修复方式
- 禁止顺手重构、统一风格或修改无关逻辑
- 禁止多个独立改动点混在一次修复里静默推进
- 禁止不做回归就宣称修复完成
- 禁止把数据补救或运行态恢复写成“代码已修好所以问题已完成”
- 禁止以"太简单"为由跳过方案确认步骤
- 禁止不写复现测试就修 Bug（除非确实无法自动化测试，需显式说明原因）
