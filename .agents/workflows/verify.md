# /verify

当实现已完成一轮，准备判断“是不是做对了、是不是真的能用”时触发。

> 路径说明：lane 布局下，下文提到的 `MISSION.md`、`DESIGN.md`、`acceptance.yaml`、`STATE.md`、`verification-matrix.yaml`、`evals/`、`specs/` 默认位于当前 lane 的 `.ai-os/lanes/<lane-id>/`；legacy 单交付项目仍位于 `.ai-os/` 根层。

## 目标

把质量拆成 4 个必须过门：

1. 设计确认门
2. 逻辑确认门
3. 实现质量门
4. 交付质量门

`reverse-spec` 项目额外检查 `parity-gate`。

## 触发条件

- 用户显式输入 `/verify`
- `/build` 阶段所有 wave 完成后自动进入
- IDE 计划模式、todo 列表或其他外部编排机制的所有任务完成后自动进入
- 无论哪种触发方式，本阶段的所有步骤和门禁要求不可削减

## 必做步骤

1. 先明确本轮验证的 lane 范围：
   - 默认只验证当前 lane
   - 若共享代码、共享契约或共享基础设施改动可能影响其他 lane，先列出受影响 lane，不要只写“全局已验证”
   - 对每条受影响 lane，明确是已执行 `validate` / `gate verify` / `release-check --lane <lane-id>`，还是仍待补回归
2. 读取 `.ai-os/acceptance.yaml`
3. 对照 `.ai-os/MISSION.md`、`.ai-os/DESIGN.md`、specs 和当前实现
4. 按需求点和验收标准逐项检查：
   - 页面 / IA / 交互是否符合 Design
   - 逻辑契约、状态流转、异常处理是否正确
   - 契约基准、字段映射 / 适配说明和实现是否一致
   - shared layer / 通用抽象改动是否已留下副作用影响清单，且清单与真实受影响模块、接口 / 页面、排除场景一致
   - 复用共享抽象、共享审计字段、统一包装层或 client/server entrypoint 的地方，是否已核对真实 schema / route / wrapper 契约，并对照同仓正常实现
   - 若存在动态路径（如 `/{id}`、`/{name}`），是否已检查常见静态子路径不会被误命中或吞掉
   - MISSION / DESIGN 中确认的技术选型（依赖名、版本约束、运行时版本等）是否与实际构建配置（POM、package.json、go.mod、Cargo.toml 等）逐项一致；出现不一致时视为阻塞项
   - brownfield / change 任务中的共享基础设施约定是否已被审计并与实现一致
   - 关键用户任务 / 操作员任务 / 系统任务是否真实可达、可执行、可完成，而不是静态占位
   - 正常路径之外，异常 / 空值 / 权限拒绝 / 超时 / 部分失败路径是否可恢复
   - 项目原生静态校验是否已执行并通过；若仓库没有入口，是否已显式记录风险或 blocker
   - IDE 内置诊断（ReadLints / 编辑器内联提示）不等同于项目原生静态校验；必须使用项目构建工具链（编译命令、类型检查命令或构建命令）生成校验证据
   - 测试、review、回归、运行态证据是否齐全
   - 未实现、未验证或仅 demo / placeholder 的能力是否被误写成完成
   - fallback 证据是否被误当成正式交付证据
5. 明确本轮哪些动作已由 AI 完成，哪些仍需人工执行（如 SQL、重启、补数、环境变更），避免把待人工动作写成已完成；同时把结论拆成代码状态 / 数据状态 / 运行状态，避免把三者混成一句“已完成”
6. **零回归门**：运行项目完整测试套件，与 `/build` 阶段记录的测试基线对比；任何从"通过"变为"失败"的测试都视为回归；存在回归时，必须先输出回归清单、影响范围和修复方案，修复并通过后才能继续
7. **模式一致性检查**：对照 `.ai-os/CONVENTIONS.md`（如存在）抽查本轮实现是否引入了不一致的代码模式（命名、错误处理、导入方向等）
8. 若存在未通过项，先输出问题清单、影响范围、整改方案和是否需要升级回 `/change-request` 或 `/debug`
9. 若本轮暴露了新的稳定 failure mode、关键 tricky path 或高频回归入口，将最小复现、放行条件和验证方法同步到 `.ai-os/evals/`、`.ai-os/verification-matrix.yaml` 或等价工件
10. 更新 `.ai-os/acceptance.yaml`
11. 更新 `.ai-os/STATE.md`

## 输出

- 更新后的 `.ai-os/acceptance.yaml`
- 逐项需求 / 验收结论摘要
- review / parity / runtime 证据摘要
- lane 范围结论：当前验证覆盖了哪些 lane，哪些 lane 仍待补回归
- **回归对比结论**（基线测试数 / 当前通过数 / 回归数）
- 模式一致性检查结论（如存在 CONVENTIONS.md）
- 新增或更新的 failure-mode guard 结论（如存在）
- 代码状态 / 数据状态 / 运行状态结论
- 对正常路径、degraded-path、回归范围、高风险专项审查，以及 `AI 已完成` / `需人工执行` 的结论

## 禁止事项

- 禁止只看代码通过就判定完成
- 禁止共享代码改动只验证当前 lane，却对其他受影响 lane 给出笼统“已验证”结论
- 禁止页面看起来像但逻辑错仍然放行
- 禁止逻辑正确但设计和信息架构明显跑偏仍然放行
- 禁止只验证正常流程，忽略异常 / 空值 / 权限拒绝 / 超时路径
- 禁止 shared layer / 通用抽象改动没有副作用清单和 parity 结论就放行
- 禁止 client/server entrypoint 变更未核对真实 route / wrapper / schema 契约就放行
- 禁止缺少项目原生静态校验证据仍然放行，除非已显式记录仓库无入口且由用户接受该限制
- 禁止存在假入口、占位流程或未验证能力仍然放行
- 禁止仍需人工执行的 SQL / 重启 / 补数 / 环境变更被写成 AI 已全部完成
- 禁止把代码状态、数据状态和运行状态混成一句模糊的“已修复 / 已完成”
- 禁止发现问题后静默整改并跳过重新验证
- 禁止发现新的稳定 failure mode 后只在当前会话里说明，不把最小复现或回归 guard 落到工件
- 禁止存在回归时跳过修复直接放行
